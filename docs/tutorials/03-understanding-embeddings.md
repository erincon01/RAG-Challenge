## What is an embedding?

An embedding is a list of 1536 floating-point numbers that represents the "meaning" of a text. Texts about similar topics produce similar vectors.

Example: "Messi scored a goal" and "Lionel Messi put the ball in the net" would have very similar embeddings, even though the words are different. "The weather is sunny today" would have a very different embedding.

## How text becomes a vector

Pipeline:

```
"Who scored?"  ->  OpenAI API  ->  [0.023, -0.041, 0.078, ..., 0.012]
                   (text-embedding-3-small)    1536 float values
```

The model (`text-embedding-3-small`) was trained on billions of text examples to learn that:

- Words about football are close to each other
- Goals, shots, scores cluster together
- Defensive actions (tackles, blocks) form another cluster

## The embedding pipeline in this project

```
StatsBomb events -> 15-second aggregation -> Summary text -> OpenAI embedding -> DB column
                    (717 rows per match)     (gpt-4o-mini)   (1536 dims)        (VECTOR type)
```

Each match is divided into 15-second buckets. Each bucket gets:

1. A JSON aggregation of all events in that 15-second window
2. A narrative summary generated by GPT-4o-mini
3. A 1536-dimensional embedding of that summary

When you search, your question is also embedded, and the system finds the 15-second buckets whose summaries are most similar to your question.

## Check embedding coverage

```bash
curl -s "http://localhost:8000/api/v1/embeddings/status?source=postgres"
```

Response shows:

- `total_rows`: 717 (both seed matches combined)
- `coverage.text-embedding-3-small`: 717 (100% covered)
- `status.done`: 717, `pending`: 0, `error`: 0

You can also see this in the UI at http://localhost:5173/embeddings

## The seed data shortcut

The seed dataset in `data/seed/` includes pre-computed embeddings, so new developers don't need an OpenAI key. The files:

- `summaries.jsonl` -- narrative text for each 15-second bucket
- `embeddings_t3_small.jsonl` -- pre-computed 1536-dim vectors

This means `make seed` loads everything including embeddings -- no API calls needed.

## Why 1536 dimensions?

The `text-embedding-3-small` model produces 1536-dimensional vectors. This is a design choice by OpenAI:

- More dimensions = more nuance in meaning
- Fewer dimensions = faster search, less storage
- 1536 is a good balance for most use cases

Each dimension captures some abstract aspect of meaning -- there's no human-interpretable label for dimension #742, but collectively they encode the semantic content.

## Storage in the database

PostgreSQL (pgvector):

```sql
summary_embedding_t3_small VECTOR(1536)
-- HNSW index for fast search
CREATE INDEX ... USING hnsw (summary_embedding_t3_small vector_cosine_ops)
```

SQL Server 2025:

```sql
embedding_3_small VECTOR(1536)
-- No HNSW index in Express edition -- uses sequential scan
```

## Adding more matches

To embed a new match (requires OpenAI key):

```bash
curl -X POST http://localhost:8000/api/v1/ingestion/full-pipeline \
  -H "Content-Type: application/json" \
  -d '{
    "source": "postgres",
    "match_ids": [3943077],
    "competition_id": 55,
    "season_id": 282
  }'
```

Cost: ~$0.15 per match with gpt-4o-mini + text-embedding-3-small.

## Key takeaways

1. Embeddings convert text meaning into numbers
2. Similar meanings produce similar vectors with low distance scores
3. The project uses 15-second match buckets as the unit of retrieval
4. Pre-computed seed data means zero API cost for getting started
5. PostgreSQL has HNSW indexes for fast search; SQL Server Express uses sequential scan
